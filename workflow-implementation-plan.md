# 工作流实现计划

## 当前判断

结合你反馈的现状，当前项目最需要优先解决的不是“继续加功能”，而是先把数据归属和状态流理顺：

1. 上传图片蓝框里的跳转链接和店铺数据不一致
2. 目前没有用户登录系统，导致“添加店铺”“品类管理”等页面像临时状态，缺少稳定归属
3. 自动刷新和提醒能力还没有形成闭环，后续扩展会受前两个问题影响

## 推荐推进顺序

### 第一阶段：先修复数据错配

目标是确保图片、店铺、展示卡片之间的绑定关系完全正确。

- 排查图片记录、店铺记录、展示卡片之间的关联字段
- 检查是否存在复用旧数据、索引错位、缓存未刷新、详情卡片绑定错误等问题
- 给关键数据增加唯一标识、来源 ID、更新时间等字段，方便排查
- 增加回归验证：新增店铺、编辑店铺、刷新页面、切换店铺、重新上传图片

### 第二阶段：接入用户系统

目标是让所有业务状态都绑定到用户，而不是停留在页面级临时状态。

- 增加登录和会话保持
- 将店铺、品类、提醒规则、图片识别结果与用户 ID 关联
- 改造“添加店铺”“品类管理”“提醒设置”相关接口，让服务端支持用户隔离
- 把首页和管理页的初始化方式改成“基于用户加载”

### 第三阶段：增强稳定性

目标是让系统在真实使用中更稳定、更容易排障。

- 完善接口错误处理、重试、超时与空状态兜底
- 区分 loading、stale、refresh 等状态，避免用户误判
- 增加日志与埋点，定位错配、加载慢、刷新失败等问题的来源

### 第四阶段：实现自动定时刷新与提醒闭环

目标是形成从数据刷新到提醒处理的完整链路。

- 设计按用户、店铺、品类维度执行的定时刷新任务
- 记录任务状态、最近成功时间、失败原因
- 增加提醒生成、去重、确认、忽略、归档流程
- 配置刷新频率、静默时间段和阈值，避免过度打扰

## 可直接开工的任务拆分

### 1. 数据绑定修复

- 检查图片记录与店铺记录的关联字段是否一致
- 检查展示卡片渲染是否使用了正确的当前店铺
- 给关键对象补充唯一标识和来源字段
- 修复可能存在的缓存或列表复用问题

### 2. 用户系统接入

- 增加登录态与会话保持
- 建立用户维度的数据隔离
- 调整接口入参与返回结构
- 确保添加店铺、品类管理、提醒设置都能落到用户下

### 3. 自动化与提醒闭环

- 定义定时刷新任务模型
- 增加任务状态与执行记录
- 实现提醒去重、确认、忽略和归档流程
- 形成“刷新 → 发现变化 → 提醒 → 处理 → 归档”的闭环

## 依赖关系

- 数据正确性优先于其他所有功能
- 用户系统是后续所有持久化能力的前提
- 定时任务和提醒必须依赖正确的数据模型和用户归属

## 里程碑建议

### 第 1 阶段

先修复链接错配，保证添加店铺后图片卡片和店铺数据一一对应。

### 第 2 阶段

接入用户登录与会话，把页面状态改成用户级持久化。

### 第 3 阶段

增加定时刷新任务、任务状态跟踪与失败重试。

### 第 4 阶段

补监控、日志、告警和长期运行机制，再继续优化性能和交互。

## 结论

如果现在只能先做一件事，建议先修复“上传图片蓝框链接与店铺不一致”的数据错配问题；如果要做下一步架构升级，优先补用户系统。
